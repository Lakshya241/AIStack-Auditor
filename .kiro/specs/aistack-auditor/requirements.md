# Requirements Document

## Introduction

AIStack Auditor is a production-quality SaaS MVP that helps startups and small teams identify overspending on AI tools (ChatGPT, Claude, Cursor, GitHub Copilot, Gemini, Windsurf). Users input their current AI subscriptions, and the platform runs a deterministic audit engine to surface redundancies, recommend cheaper plans or alternative tools, calculate monthly and yearly savings, and generate an AI-powered summary. The platform captures leads via a form, stores them in Supabase, sends a confirmation email via Resend, and creates public shareable audit report URLs with Open Graph metadata. There is no authentication — the product is fully public and frictionless.

---

## Glossary

- **Audit_Engine**: The deterministic, rule-based module that evaluates a user's AI tool stack and produces recommendations and savings calculations.
- **Audit_Form**: The multi-row UI form where users input their AI tools, plans, seat counts, monthly spend, and use cases.
- **Audit_Report**: The results page rendered at `/report/[id]` showing savings, recommendations, optimization score, AI summary, and lead capture form.
- **Lead**: A record containing a user's email, company name, role, and team size, stored in Supabase after the user submits the lead capture form.
- **Recommendation**: A structured output from the Audit_Engine suggesting a cheaper plan or alternative tool, with a calculated savings amount and explanation.
- **Optimization_Score**: A numeric score (0–100) computed from the ratio of potential savings to total current spend, representing how optimized the user's AI stack is.
- **AI_Summary**: A natural-language summary of the audit results (80–120 words) generated by the Claude API, with a deterministic fallback if the API is unavailable.
- **Shareable_Report**: A public URL at `/report/[id]` that renders the Audit_Report for a given audit ID, accessible without authentication.
- **Tool**: One of the supported AI products: ChatGPT, Claude, Cursor, GitHub Copilot, Gemini, or Windsurf.
- **Plan**: A named subscription tier for a Tool (e.g., Free, Plus, Team, Enterprise, Pro, Business).
- **Pricing_Data**: Hardcoded, versioned per-seat monthly pricing for each Tool and Plan combination.
- **Overlap**: A condition where two or more Tools in a user's stack serve substantially the same use case, flagged by the Audit_Engine as redundant.
- **Resend**: The third-party email delivery service used to send lead confirmation emails.
- **Supabase**: The hosted Postgres-backed backend used for persisting audit records and leads.

## Requirements

### Requirement 1: AI Tool Stack Input

**User Story:** As a startup founder or team lead, I want to input all the AI tools my team currently pays for — including plan, seat count, and monthly spend — so that the platform can audit my spending accurately.

#### Acceptance Criteria

1. THE Audit_Form SHALL render a dynamic list of tool-entry rows, each containing fields for Tool name, Plan, number of seats, monthly spend (USD), and use case description (maximum 500 characters).
2. THE Audit_Form SHALL allow users to add up to 10 tool rows; the remove button for a row SHALL be disabled (not hidden) when only 1 row remains.
3. WHEN a user selects a Tool name, THE Audit_Form SHALL populate a dropdown of valid Plans for that Tool sourced from Pricing_Data; IF the selected Tool has no Plans defined in Pricing_Data, THE Audit_Form SHALL display an error message on that row indicating the tool configuration is unavailable.
4. WHEN a user submits the Audit_Form with one or more rows where required fields (Tool, Plan, seats, monthly spend) are empty, THE Audit_Form SHALL display inline validation errors for each invalid field and prevent submission.
5. WHEN a user enters a monthly spend value less than 0 or greater than 1,000,000, THE Audit_Form SHALL display a validation error on that field.
6. WHEN a user enters a seat count less than 1 or greater than 10,000, THE Audit_Form SHALL display a validation error on that field.
7. WHILE a user is filling out the Audit_Form, THE Audit_Form SHALL persist the current form state to localStorage so that a page refresh does not lose entered data.
8. WHEN a user returns to the Audit_Form after a previous session, THE Audit_Form SHALL restore the previously saved form state from localStorage; IF the stored data is unrecognizable or corrupt, THE Audit_Form SHALL discard it and render a single empty tool-entry row.
9. THE Audit_Form SHALL render without horizontal scrollbars and all interactive elements SHALL be reachable via keyboard navigation on viewport widths from 320px to 2560px.

### Requirement 2: Deterministic Audit Engine

**User Story:** As a user, I want the platform to analyze my AI tool stack using consistent, rule-based logic so that I receive reliable and reproducible recommendations every time I submit the same data.

#### Acceptance Criteria

1. THE Audit_Engine SHALL evaluate each submitted Tool entry against a rule set identified by a unique version string to produce zero or more Recommendations.
2. WHEN a user submits a ChatGPT Team plan with 2 or fewer seats, THE Audit_Engine SHALL produce a Recommendation to downgrade to ChatGPT Plus with monthly savings calculated as (ChatGPT Team price − ChatGPT Plus price) × seat count.
3. WHEN a user submits an Enterprise plan for any Tool with fewer than 5 seats, and a lower plan tier exists for that Tool in Pricing_Data, THE Audit_Engine SHALL produce a Recommendation to downgrade to the next lower plan tier with monthly savings calculated as (Enterprise price − next lower plan price) × seat count.
4. WHEN a user's stack contains ChatGPT and Claude and Gemini simultaneously, THE Audit_Engine SHALL produce a single Overlap Recommendation that identifies all three tools by name as redundant general-purpose LLM assistants.
5. WHEN a user's stack contains both Cursor and GitHub Copilot, THE Audit_Engine SHALL produce an Overlap Recommendation identifying both tools by name as redundant AI coding assistants and recommending retention of the tool with the higher seat count; IF seat counts are equal, THE Audit_Engine SHALL recommend retaining either tool.
6. WHEN a user's stack contains both Cursor and Windsurf, THE Audit_Engine SHALL produce an Overlap Recommendation identifying both tools by name as redundant AI IDE tools and recommending retention of the tool with the higher seat count; IF seat counts are equal, THE Audit_Engine SHALL recommend retaining either tool.
7. WHEN a user submits a paid plan for any Tool where the number of seats is 1, and a Free tier exists for that Tool in Pricing_Data, THE Audit_Engine SHALL produce a Recommendation to switch to the Free tier with monthly savings equal to the user's current monthly spend for that Tool entry.
8. THE Audit_Engine SHALL be a pure function: given the same input array of Tool entries, THE Audit_Engine SHALL always return the identical array of Recommendations in the same order with the same savings figures.
9. THE Audit_Engine SHALL complete its evaluation synchronously in under 50ms for any input with up to 10 Tool entries.
10. THE Audit_Engine SHALL NOT call any external API or perform any I/O; all logic SHALL be self-contained within `lib/audit/`.
11. WHEN the Audit_Engine receives an input containing a Tool entry with a missing or invalid required field (Tool name, Plan, seats, or monthly spend), THE Audit_Engine SHALL return an error result identifying the invalid entry rather than producing partial Recommendations.
12. WHEN the Audit_Engine receives an input array containing more than 10 Tool entries, THE Audit_Engine SHALL return an error result indicating the input exceeds the maximum allowed size.

### Requirement 3: Savings Calculations

**User Story:** As a user, I want to see exactly how much money I could save monthly and annually by following the recommendations, so that I can make an informed decision about changing my subscriptions.

#### Acceptance Criteria

1. THE Audit_Engine SHALL calculate the monthly savings for each plan-downgrade Recommendation as (current plan price per seat × seat count) − (recommended plan price per seat × seat count), where plan prices are sourced exclusively from Pricing_Data.
2. THE Audit_Engine SHALL calculate the yearly savings for each Recommendation as the monthly savings for that Recommendation multiplied by 12.
3. THE Audit_Report SHALL display the total potential monthly savings as the sum of monthly savings across all Recommendations.
4. THE Audit_Report SHALL display the total potential yearly savings as the sum of yearly savings across all Recommendations.
5. WHEN a Recommendation involves an Overlap (redundant tools), THE Audit_Engine SHALL calculate savings as the sum of the full monthly spend values of the Tool entries recommended for removal.
6. THE Audit_Engine SHALL express all monetary values in USD, rounded to two decimal places using standard mathematical rounding (half up).
7. IF the calculated savings for any individual Recommendation is zero or negative, THEN THE Audit_Engine SHALL omit that Recommendation from the output; Recommendations with any positive savings value, however small, SHALL be included.

### Requirement 4: Optimization Score

**User Story:** As a user, I want a single score that tells me how well-optimized my AI spending is, so that I can quickly understand the overall health of my stack.

#### Acceptance Criteria

1. THE Audit_Engine SHALL compute the Optimization_Score as an integer between 0 and 100, where 100 represents a fully optimized stack with no identified savings.
2. THE Audit_Engine SHALL derive the Optimization_Score using the formula: `round((1 - totalMonthlySavings / totalCurrentSpend) * 100)`, where `totalCurrentSpend` is the sum of all user-entered monthly spend values, `totalMonthlySavings` is the sum of monthly savings across all Recommendations, and `round` applies standard mathematical rounding (half up); the result SHALL be clamped to the range [0, 100].
3. IF the total current spend across all submitted Tool entries is zero, THEN THE Audit_Engine SHALL return an Optimization_Score of 100, treating the zero-spend situation as fully optimized regardless of any theoretical savings.
4. THE Audit_Report SHALL display the Optimization_Score as a numeric value accompanied by a color-coded label rendered in the same visual container: green for scores 80–100, yellow for 50–79, and red for 0–49.

### Requirement 5: Recommendations Display

**User Story:** As a user, I want to see clear, actionable recommendation cards on my report so that I know exactly what to change and why.

#### Acceptance Criteria

1. THE Audit_Report SHALL render one card per Recommendation, each displaying the Tool name, the current plan, the recommended plan or action, the monthly savings formatted as a USD currency value, the yearly savings formatted as a USD currency value, and an explanation string produced by the Audit_Engine describing the reason for the Recommendation.
2. IF there are no Recommendations, THEN THE Audit_Report SHALL display a message indicating the user's stack is already well-optimized.
3. THE Audit_Report SHALL sort Recommendation cards in descending order of monthly savings; cards with equal monthly savings SHALL be sorted by Tool name in ascending alphabetical order as a secondary sort key.
4. THE Audit_Report SHALL display the current plan's monthly cost per seat and the recommended plan's monthly cost per seat side by side for plan-downgrade Recommendations.
5. THE Audit_Report SHALL display an Overlap badge on Recommendation cards that were generated from an Overlap rule; Recommendation cards generated from plan-downgrade or free-tier rules SHALL NOT display an Overlap badge.

### Requirement 6: AI-Powered Summary

**User Story:** As a user, I want a concise natural-language summary of my audit results so that I can quickly understand the key findings without reading every card.

#### Acceptance Criteria

1. WHEN an audit is submitted, THE AI_Summary module SHALL call the Claude API using the prompt template defined in `lib/ai/prompts.ts` and SHALL target a summary between 80 and 120 words.
2. THE AI_Summary module SHALL NOT embed prompt strings directly in API route handlers; all prompt construction SHALL occur exclusively within `lib/ai/prompts.ts`.
3. IF the Claude API call fails or returns an error, THEN THE AI_Summary module SHALL return a deterministic fallback summary constructed from the Recommendations data that includes the total savings figure and the count of Recommendations, without calling any external service; IF the fallback itself fails to generate a summary, THEN THE AI_Summary module SHALL return an error result.
4. THE AI_Summary module SHALL complete its response within 10 seconds; IF the Claude API does not respond within 10 seconds, THEN THE AI_Summary module SHALL use the fallback summary; IF the fallback also fails, THEN THE AI_Summary module SHALL return an error result.
5. THE Audit_Report SHALL display the AI_Summary in a card rendered with a distinct background color and a visible border, labeled "AI Insights"; IF the AI_Summary module returns an error result, THE Audit_Report SHALL hide the AI Insights card and display an inline error message in its place.
6. THE AI_Summary SHALL describe findings and savings opportunities only; THE AI_Summary SHALL NOT make specific product recommendations that contradict the Audit_Engine's deterministic output.

### Requirement 7: Lead Capture

**User Story:** As a business operator, I want to capture the email, company, role, and team size of users who view their audit report so that I can follow up with them for sales and marketing purposes.

#### Acceptance Criteria

1. THE Audit_Report SHALL render a lead capture form with fields for email address (required), company name (required), role (required), and team size (required, numeric, minimum 1, maximum 10,000).
2. WHEN a user submits the lead capture form with all required fields valid — where a valid email address conforms to RFC 5322 format — THE Lead_Capture module SHALL store a Lead record in Supabase containing the email, company name, role, team size, audit ID, and submission timestamp.
3. WHEN a user submits the lead capture form with any required field missing or invalid, THE Lead_Capture module SHALL display an inline validation error on each invalid field and prevent submission.
4. WHEN a Lead is successfully stored, THE Lead_Capture module SHALL trigger Resend to send a confirmation email to the provided address containing the shareable report URL.
5. WHEN a Lead is successfully stored, THE Audit_Report SHALL replace the lead capture form with a confirmation state that displays the shareable report URL as a visible, copyable link.
6. IF the Supabase write operation fails, THEN THE Lead_Capture module SHALL display an error message indicating the save failed and SHALL NOT send a confirmation email.
7. IF the Resend email delivery fails, THEN THE Lead_Capture module SHALL log the error server-side and SHALL NOT surface the email failure to the user, as the Lead record has already been saved.
8. THE Lead_Capture module SHALL validate all lead fields against a Zod schema before performing any database write; no field-level validation logic SHALL exist outside of that Zod schema.

### Requirement 8: Shareable Audit Report URLs

**User Story:** As a user, I want to share my audit report with my team or investors via a public URL so that others can view the findings without needing to re-enter data.

#### Acceptance Criteria

1. WHEN an audit is submitted, THE Share module SHALL generate a unique audit ID using UUID v4 and attempt to persist the full audit payload (Tool entries, Recommendations, savings totals, Optimization_Score, AI_Summary) to Supabase; IF the Supabase write fails, THE Share module SHALL return an error result and SHALL NOT return an audit ID.
2. THE Share module SHALL make the Shareable_Report accessible at the public URL path `/report/[id]` without requiring authentication.
3. WHEN a valid audit ID is requested at `/report/[id]`, THE Audit_Report page SHALL retrieve the audit data from Supabase and render the full report.
4. WHEN an invalid or non-existent audit ID is requested, or when any data retrieval failure occurs (including temporary database unavailability or data corruption), THE Audit_Report page at `/report/[id]` SHALL render a 404 page with a link back to the Audit_Form.
5. THE Audit_Report page at `/report/[id]` SHALL include Open Graph meta tags with the following content: `og:title` set to "AIStack Auditor Report — Save $[totalMonthlySavings]/mo", `og:description` set to "Optimization score: [score]/100. View your AI spending audit.", and `og:url` set to the canonical report URL.
6. THE Share module SHALL generate audit IDs using a UUID v4 algorithm to ensure global uniqueness.

### Requirement 9: Landing Page

**User Story:** As a potential user, I want a clear and compelling landing page so that I can understand what AIStack Auditor does and start an audit immediately.

#### Acceptance Criteria

1. THE Landing_Page SHALL render the following sections in order: Hero, Features, How It Works, FAQ, CTA, and Footer.
2. THE Landing_Page Hero section SHALL include a headline, a sub-headline describing the value proposition, and a primary CTA button; WHEN a user clicks the CTA button, THE Landing_Page SHALL navigate to the Audit_Form.
3. THE Landing_Page Features section SHALL list at least four key capabilities of the platform, each with an icon and a description of no more than 100 characters.
4. THE Landing_Page How_It_Works section SHALL describe the audit process in three or fewer numbered steps.
5. THE Landing_Page FAQ section SHALL include at least four questions; WHEN a user clicks a question, THE Landing_Page SHALL expand or collapse the corresponding answer.
6. THE Landing_Page SHALL achieve a Lighthouse performance score of 90 or above on desktop when measured on the production Vercel deployment.
7. THE Landing_Page SHALL render without horizontal scrollbars and all interactive elements SHALL be reachable via keyboard navigation on viewport widths from 320px to 2560px.

### Requirement 10: Pricing Data Accuracy and Versioning

**User Story:** As a developer maintaining the platform, I want all pricing data to be centralized and versioned so that updates to tool pricing do not require changes scattered across the codebase.

#### Acceptance Criteria

1. THE Pricing_Data module SHALL define per-seat monthly prices for all Plans of all supported Tools in a single file at `lib/constants/pricingData.ts`.
2. THE Pricing_Data module SHALL export a `PRICING_DATA_VERSION` string constant in semver format (e.g., "1.0.0") that SHALL be incremented whenever any pricing value is updated.
3. THE Audit_Engine SHALL source all pricing figures exclusively from the Pricing_Data module and SHALL NOT hardcode monetary values inline in rule or calculation logic.
4. THE Pricing_Data module SHALL include per-seat monthly prices for the following Tools and Plans at minimum:
   - ChatGPT: Free ($0), Plus ($20), Team ($25), Enterprise ($60)
   - Claude: Free ($0), Pro ($20), Team ($25), Enterprise ($60)
   - Cursor: Free ($0), Pro ($20), Business ($40)
   - GitHub Copilot: Free ($0), Individual ($10), Business ($19), Enterprise ($39)
   - Gemini: Free ($0), Business ($20), Enterprise ($30)
   - Windsurf: Free ($0), Pro ($15), Teams ($35)
5. THE Pricing_Data module SHALL contain only constant declarations and type annotations; it SHALL NOT contain function calls, class instantiations, or any expression that produces a side effect at import time.
6. THE Pricing_Data module SHALL define all Plan entries referenced by the Audit_Engine; IF the Audit_Engine references a Tool/Plan combination not present in Pricing_Data, THE Audit_Engine SHALL return an error result rather than silently using an undefined value.

### Requirement 11: API Routes and Data Flow

**User Story:** As a developer, I want well-defined API routes that separate concerns cleanly so that the frontend, audit logic, AI integration, and data persistence are independently maintainable.

#### Acceptance Criteria

1. WHEN a request is received at `POST /api/audit`, THE platform SHALL validate the request body against the audit Zod schema, run the Audit_Engine, persist the result to Supabase, and return the audit ID, Recommendations, savings totals, and Optimization_Score; IF the Supabase persistence fails, THE route SHALL return a 503 status code and SHALL NOT return a partial audit result.
2. WHEN a request is received at `POST /api/generate-summary`, THE platform SHALL accept audit results and return an AI_Summary string, using the fallback if the Claude API is unavailable; IF both Claude and the fallback fail, THE route SHALL return a 500 status code.
3. WHEN a request is received at `POST /api/leads`, THE platform SHALL validate lead fields, write the Lead to Supabase, and trigger the Resend confirmation email.
4. WHEN a request is received at `GET /api/share/[id]`, THE platform SHALL retrieve the persisted audit record from Supabase by ID and return it as JSON; IF no record exists for the given ID, THE route SHALL return a 404 status code.
5. WHEN any API route receives a request body that fails Zod schema validation, THE API route SHALL return a 400 status code with a JSON body containing an `errors` array describing each validation failure.
6. WHEN any API route encounters an unexpected server error, THE API route SHALL return a 500 status code with a JSON body containing a `message` field and SHALL log the error server-side.
7. WHEN a request is received at `POST /api/audit` with an input containing up to 10 Tool entries, THE route SHALL complete its full response (excluding AI summary generation) within 3 seconds.

### Requirement 12: Testing

**User Story:** As a developer, I want a comprehensive automated test suite covering the audit engine, pricing rules, and calculations so that I can confidently refactor and extend the platform.

#### Acceptance Criteria

1. THE test suite SHALL include a minimum of 5 test cases implemented with Vitest, with at least one test covering the Audit_Engine module, at least one test covering the Pricing_Data module, and at least one test covering the savings calculation logic.
2. THE test suite SHALL include a test verifying that the ChatGPT Team + 2 seats rule produces a Recommendation with monthly savings of $10.00 and yearly savings of $120.00 (derived from ChatGPT Team at $25/seat minus ChatGPT Plus at $20/seat, multiplied by 2 seats).
3. THE test suite SHALL include a test verifying that the ChatGPT + Claude + Gemini Overlap rule produces an Overlap Recommendation that identifies all three tools by name.
4. THE test suite SHALL include a test verifying that the Optimization_Score formula returns 75 for a total spend of $100 and total savings of $25, and returns 100 when total spend is $0.
5. THE test suite SHALL include a test verifying that the Audit_Engine returns an empty Recommendations array when given a Tool entry that matches no rules.
6. THE test suite SHALL include a test verifying that all monetary values in Recommendations are rounded to two decimal places.
7. WHEN the test suite is executed with `vitest --run`, THE test suite SHALL complete with zero failing tests.

### Requirement 13: Code Quality and Architecture

**User Story:** As a developer, I want the codebase to follow strict TypeScript conventions and a clean separation of concerns so that the project is maintainable and extensible.

#### Acceptance Criteria

1. THE codebase SHALL use TypeScript with `strict: true` in `tsconfig.json` and SHALL contain no uses of the `any` type in any file under `app/`, `components/`, `lib/`, `hooks/`, or `types/`.
2. THE codebase SHALL define all domain types in the `types/` directory, with separate files for `audit.ts`, `tool.ts`, `recommendation.ts`, and `api.ts`.
3. THE Audit_Engine SHALL be implemented as a pure function exported from `lib/audit/engine.ts` with no imports from `next`, `react`, `react-dom`, `@radix-ui`, or any package under `components/`.
4. THE codebase SHALL contain no duplicated business logic; pricing lookups, savings calculations, and rule evaluations SHALL each exist in exactly one module.
5. THE codebase SHALL follow this exact top-level folder structure: `app/`, `components/`, `lib/`, `hooks/`, `tests/`, `types/`, `public/`; no additional top-level directories SHALL be present.
6. ALL React components SHALL be typed with a named TypeScript interface or type alias for their props; no component SHALL accept props typed as an inline object literal or left untyped.
7. THE codebase SHALL use React Hook Form with Zod resolvers for all form validation; no form SHALL contain conditional field-level validation logic that is not expressed as a rule within a Zod schema.
